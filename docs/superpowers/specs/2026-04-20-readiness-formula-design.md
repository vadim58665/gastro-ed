# Дизайн: Новая формула готовности к аккредитации

**Дата:** 2026-04-20
**Статус:** Design approved, spec на согласовании

## Контекст

На странице профиля (вкладка «Подготовка») пользователь видит набор метрик: «Готовность к аккредитации», «Точность», «N/M вопросов», «0/4 блоков», «185 ошибок», списки «слабых» и «проблемных» вопросов. Текущая логика разбросана по `PrepProfile.tsx`, `useAccreditation.ts`, `accreditationErrors.ts` и страдает от нескольких несоответствий.

### Текущие баги и несоответствия

1. **`totalLearned` (188) больше числа правильных ответов (109).**
   `block.learned++` срабатывает на каждый правильный ответ без дедупликации по `questionId`, а `block.total` захардкожен в 100 вместо реального размера блока (`useAccreditation.ts:72,75`).

2. **«Готовность 56%» измеряет только охват.**
   Формула `totalLearned / totalQuestions` игнорирует точность. Можно тыкать наугад и получить 100% готовности при точности 20%.

3. **Два разных определения «ошибки» на одном экране.**
   Одно место считает уникальные нерешённые (185), другое показывает все ошибочные попытки (~318). Пользователь путается.

4. **Нет учёта времени и забывания.**
   Правильный ответ месячной давности эквивалентен правильному ответу сегодня.

5. **Блоки классифицируются по accuracy попыток.**
   Пользователь, ответивший правильно 1 раз из 1 по блоку, получает «strong». Нет учёта уверенности.

## Цели

- Одна честная метрика готовности: прогноз шанса сдать экзамен на ≥70%.
- Учёт **времени** с последнего ответа и **серии правильных повторов**.
- Единое определение «ошибки» / «слабого вопроса».
- Детерминированный расчёт за миллисекунды на 300+ вопросах.
- Не ломать существующие данные пользователей: миграция без потерь.

## Не-цели

- Не заменяем FSRS в `useReview` (это отдельная система для ленты).
- Не добавляем серверное хранение прогресса (остаёмся на localStorage).
- Не меняем процесс прохождения тестов/экзамена.
- Не добавляем калибровку на основе результатов пробных экзаменов (Phase 2, не входит в этот spec).

## Дизайн

### 1. Модель памяти одного вопроса

Для каждого вопроса считаем `p_correct(q)` — вероятность того, что пользователь правильно ответит прямо сейчас.

```
если state.attempts == 0:
    p_correct = 0.25     # не видел, baseline угадывания (1 из 4 вариантов)

иначе если state.lastAnswerCorrect == false:
    p_correct = 0.25     # последний ответ неверный, сброс к baseline

иначе:
    days = (Date.now() - state.lastSeen) / 86400000
    S    = S_0 × β^(state.correctStreak - 1)
    p_correct = 0.25 + 0.75 × Math.exp(-days / S)
```

**Параметры:**

- `S_0 = 3` (базовая стабильность в днях после первого правильного ответа)
- `β = 2` (множитель стабильности за каждое следующее правильное повторение)
- `baseline = 0.25` (нижняя граница: угадывание 1 из 4 вариантов)

**Свойства:**

- `p_correct ∈ [0.25, 1.0]`. Никогда не опускается ниже baseline (медленный откат вместо резкого обрыва).
- Первый правильный ответ держится ~3 дня (`p ≈ 53%` на 3-й день).
- Каждый следующий правильный удваивает «срок годности»: 2-й правильный → S=6 дней, 3-й → 12, 4-й → 24, 5-й → 48.
- Одна ошибка откатывает к baseline (честно: значит пользователь не знает).

**Примеры значений:**

| streak | дней с показа | S | p_correct |
|:-:|:-:|:-:|:-:|
| 1 | 0 | 3 | 1.00 |
| 1 | 3 | 3 | 0.53 |
| 1 | 7 | 3 | 0.32 |
| 2 | 7 | 6 | 0.48 |
| 3 | 7 | 12 | 0.67 |
| 3 | 14 | 12 | 0.48 |
| 4 | 30 | 24 | 0.47 |
| 5 | 30 | 48 | 0.65 |
| 5 | 60 | 48 | 0.47 |

### 2. Готовность к аккредитации

Экзамен = случайная выборка **80 вопросов** из всех `N` вопросов специальности. Проходной балл ≥70% (56 из 80).

**Задача.** Для каждого вопроса известно `p_i = p_correct(q_i)`. Случайно выбирается 80 из N, каждый даёт независимо Бернулли(p_i) правильный ответ. Нужно `P(sum >= 56)` усреднённое по случайным выборкам.

**Подход: нормальное приближение (CLT).**

Случайная выборка 80 вопросов из большого N с последующими независимыми бросками даёт сумму правильных ответов X с параметрами:

```
μ  = 80 × avg(p_i)
σ² = 80 × avg(p_i × (1 - p_i)) + 80 × (1 - 80/N) × var(p_i)
   ≈ 80 × avg(p_i × (1 - p_i))   // второй член мал при N >> 80, им пренебрегаем
```

С поправкой на непрерывность (continuity correction):

```
P(сдать) = 1 - Φ((55.5 - μ) / σ)
```

где `Φ` — стандартная нормальная CDF (реализация через `erf`, ~15 строк).

**Почему не Poisson Binomial DP:** PB требует фиксированного списка из 80 конкретных вероятностей, но экзамен выбирается случайно. Строгий учёт комбинаторики выборки сложнее нормальной аппроксимации и не даёт выигрыша в точности при N=336.

**Почему не Monte Carlo:** случайная симуляция не детерминирована (результат колеблется ±1-2% между пересчётами при 500 итерациях), что создаёт иллюзию движения числа. Нормальное приближение стабильно.

**Точность приближения:** при `n=80` и `p` в диапазоне [0.25, 1.0] CLT даёт ошибку <1% по сравнению с точным расчётом. Достаточно для UI.

Псевдокод:

```typescript
function examReadiness(probs: number[], drawSize = 80, passThreshold = 0.7): number {
  if (probs.length === 0) return 0;
  const minCorrect = Math.ceil(drawSize * passThreshold); // 56
  const avgP = mean(probs);
  const avgVarP = mean(probs.map(p => p * (1 - p)));
  const mu = drawSize * avgP;
  const sigma = Math.sqrt(drawSize * avgVarP);
  if (sigma === 0) return mu >= minCorrect ? 1 : 0;
  const z = (minCorrect - 0.5 - mu) / sigma;
  return 1 - normalCdf(z);
}

function normalCdf(z: number): number {
  return 0.5 * (1 + erf(z / Math.SQRT2));
}

function erf(x: number): number {
  // Abramowitz & Stegun 7.1.26, max error ≈ 1.5e-7
  const sign = Math.sign(x);
  const ax = Math.abs(x);
  const t = 1 / (1 + 0.3275911 * ax);
  const y = 1 - (((((1.061405429 * t - 1.453152027) * t) + 1.421413741) * t - 0.284496736) * t + 0.254829592) * t * Math.exp(-ax * ax);
  return sign * y;
}
```

Сложность O(N), пара проходов по массиву `probs`. На 336 вопросах ~0.1 мс.

### 3. UI-метрики (новые три плитки вместо текущей мешанины)

| Название | Что значит | Формула |
|---|---|---|
| **Готовность** | Шанс сдать пробный экзамен на ≥70% | `examReadiness(probs, 80, 0.7) × 100%` |
| **Охват** | Сколько вопросов хотя бы раз встречали | `count(q : stats[q].attempts > 0) / N` |
| **Средняя прочность** | Средняя вероятность знать случайный вопрос | `avg(p_correct)` |

Вспомогательные:

- **Слабые вопросы**: `count(q : p_correct(q) < 0.4)`. Заменяет «185 ошибок».
- **До зелёной зоны**: число вопросов, которое нужно «поднять» до `p_correct ≥ 0.7`, чтобы готовность достигла 70%. Пересчёт: берём текущий `P(сдать)`, бинарным поиском находим минимальное число «вылеченных» слабых вопросов.

### 4. Классификация блоков

Считается по среднему `p_correct` внутри блока:

```
avg_p = avg(p_correct(q) для q в блоке)
coverage = count(q : stats[q].attempts > 0) / block_size

уровень блока:
  not_started  если coverage == 0
  started      если coverage < 0.2
  weak         если avg_p < 0.4
  ready        если avg_p >= 0.4 И avg_p < 0.85
  strong       если avg_p >= 0.85 И coverage >= 0.8
```

«Пройдено блоков» = `ready + strong`.

### 6. Миграция существующих данных

В `useAccreditation.ts` при загрузке прогресса из localStorage:

```ts
function migrateStats(raw: QuestionStats): QuestionStats {
  if (typeof raw.lastAnswerCorrect === 'boolean' && typeof raw.correctStreak === 'number') {
    return raw; // уже мигрировано
  }
  // Грубая оценка: если когда-то отвечал правильно, считаем что последний тоже правильный
  return {
    ...raw,
    lastAnswerCorrect: raw.wasEverCorrect,
    correctStreak: raw.wasEverCorrect ? 1 : 0,
  };
}
```

Миграция не пересчитывает историю (её нет). За 2-3 сессии новой работы данные откалибруются по факту.

### 7. Удаление legacy-полей

- `BlockProgress.learned` и `BlockProgress.total` — удалить из типа (или оставить deprecated и не использовать).
- `markQuestionLearned` в `useAccreditation.ts` — удалить. Все места вызова (`tests/[blockId]/page.tsx:47`, `modes/exam/ExamInner.tsx:151`) переходят на `recordAnswer` (уже вызывается рядом).
- Счётчик `totalLearned` — удалить, заменить на `coverageCount = count(q : stats[q].attempts > 0)`.

### 8. Обновление `recordAnswer`

```ts
const recordAnswer = (questionId: string, correct: boolean) => {
  updateProgress((prev) => {
    const stats = { ...(prev.questionStats ?? {}) };
    const existing = stats[questionId] ?? {
      attempts: 0, wrong: 0, lastSeen: 0,
      wasEverCorrect: false, lastAnswerCorrect: false, correctStreak: 0,
    };
    stats[questionId] = {
      attempts: existing.attempts + 1,
      wrong: existing.wrong + (correct ? 0 : 1),
      lastSeen: Date.now(),
      wasEverCorrect: existing.wasEverCorrect || correct,
      lastAnswerCorrect: correct,
      correctStreak: correct ? existing.correctStreak + 1 : 0,
    };
    // mistakes[] остаётся как есть (используется для режима «повторить ошибки»)
    const mistakes = correct
      ? prev.mistakes.filter((id) => id !== questionId)
      : (prev.mistakes.includes(questionId) ? prev.mistakes : [...prev.mistakes, questionId]);
    return { ...prev, mistakes, questionStats: stats, updatedAt: Date.now() };
  });
};
```

## Edge cases

1. **Пустой прогресс (`attempts: 0` для всех).** `avg(p_correct) = 0.25`, `P(сдать экзамен ≥70%)` практически 0 (биномиальный тест 80 бросков с p=0.25 и k≥56 даёт ~10⁻¹⁹). Это корректно.
2. **Пользователь только что сдал идеально все 336 вопросов.** Все `p_correct ≈ 1.0`, `P(сдать) ≈ 100%`. Корректно.
3. **Забывание.** Если 30 дней не заходил, `days_since_seen` большой, `p_correct` скатывается к 0.25 даже с высоким streak. Готовность уменьшится. Это и нужно.
4. **Единичная ошибка в старом материале.** `correctStreak → 0`, `lastAnswerCorrect → false`. Следующий правильный ответ поднимет streak до 1, `S = 3`. Мягкий откат, не трагедия.
5. **Специальность без вопросов.** `N = 0`, ранний return: готовность 0%, охват 0/0, средняя прочность 0%.
6. **Размер блока ≠ 100.** Уже учтено: блок классифицируется по реальным вопросам блока, а не по хардкоду 100.
7. **Вопрос без `questionStats` (старый дамп).** Миграция обрабатывает; если даже после миграции поле отсутствует — `pCorrect` возвращает `BASELINE`.
8. **Экзамен выбирает `drawSize > N`.** Клип к `min(drawSize, N)`. На практике 80 < 300+, не актуально, но guard на всякий случай.
9. **Детерминированность.** `pCorrect` зависит от `Date.now()`. Это корректно — состояние памяти реально меняется со временем. Для тестов инжектируем `now` параметром.
10. **Численная устойчивость.** Формула использует `avg`, `exp`, `erf` — всё стабильно в Float64 для ожидаемых диапазонов (z-value редко выходит за [-10, 10]).

## Тестовая стратегия

Vitest, `src/__tests__/readiness.test.ts`. Ключевые проверки:

- `pCorrect` на 10+ точках таблицы значений (сверяемся с ожидаемыми, tolerance 0.01).
- `examReadiness` при одинаковых `p_i` совпадает с биномиальной CDF (tolerance 0.02 из-за CLT-аппроксимации).
- Поведение границ: все `p_i = 0` → готовность ≈ 0; все `p_i = 1` → готовность ≈ 1.
- `classifyBlock` на пяти уровнях: not_started, started, weak, ready, strong.
- Миграция `QuestionStats` на данных без новых полей: старые структуры превращаются в валидные, сохраняется поведение.

## План развёртывания

1. Изменения полностью локальные (localStorage), откат безопасен.
2. При первом запуске после деплоя миграция срабатывает автоматически.
3. Пользователи увидят новые числа сразу. Готовность, скорее всего, **снизится** относительно текущего «охват-only» показателя (у пользователя на скриншоте реальная готовность будет ~5%, а не 56%). Это честнее.
4. Коммуникация пользователям не требуется (это не breaking feature).

## Phase 2 (вне этого spec)

- Калибровка параметров `S_0`, `β` на реальных данных пробных экзаменов.
- Учёт категорий сложности (difficulty 1-5) как веса в recall.
- Интеграция с FSRS: вместо собственной кривой переиспользовать `useReview` state.
- Визуализация прогноза на графике (готовность по дням до экзамена).

## Открытые вопросы

Нет. Все параметры зафиксированы: `S₀=3`, `β=2`, `baseline=0.25`, `drawSize=80`, `passThreshold=0.7`, `weakThreshold=0.4`, пороги блоков `weak<0.4, ready[0.4,0.85), strong>=0.85 & coverage>=0.8`.
